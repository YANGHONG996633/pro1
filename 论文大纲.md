### 第二章 系统总体设计

- 2.1 系统需求分析
  - 功能需求（实时检测、自动分拣、串口通信）
  - 性能需求（推理延迟 < 300ms、通信超时容错）
  - 硬件约束（STM32F103C6 仅 10KB SRAM、Jetson Nano 算力预算）
- 2.2 系统总体架构
  - 感知层（USB摄像头 + 光电开关）
  - 边缘推理层（Jetson Nano + TensorRT + YOLOv8）
  - 控制执行层（STM32 + FreeRTOS + TB6600 步进电机 + 气缸）
  - 通信层（UART 自定义帧协议 + XOR 校验）
- 2.3 关键技术选型
  - MCU：STM32F103C6（Cortex-M3, 72MHz）
  - RTOS：FreeRTOS（抢占式多任务调度）
  - 边缘AI平台：NVIDIA Jetson Nano (JetPack 4.6.1)
  - 推理框架：TensorRT FP16
  - 检测模型：YOLOv8n
  - 电机驱动：TB6600 + 42步进电机
  - 执行机构：电磁阀控制气缸

### 第四章 嵌入式软件设计

- 4.1 FreeRTOS 实时操作系统配置
  - 抢占式调度策略与优先级分配（DetectTask=Normal, ControlTask=AboveNormal, UartTask=Normal）
  - 堆内存管理（`configTOTAL_HEAP_SIZE = 4800`，适配 10KB SRAM 约束）
  - 任务栈空间估算与优化（6 任务合计约 4040B heap）
- 4.2 多任务协同设计
  - vDetectTask：光电开关轮询 + 软件去抖（20ms）+ 物体离开检测
  - vControlTask：传送带启停 → 串口触发 → 等待回复 → 分拣动作
  - vUartTask：帧组包（XOR 校验）+ 阻塞接收 + 超时容错
- 4.3 任务同步机制
  - 三个信号量实现生产者-消费者模式
  - `detectSemHandle` → `uartSemHandle` → `uartDoneSemHandle` 流水线同步
  - 任务调度时序图
- 4.4 自定义串口通信协议设计
  - 上行帧：`AA 55 10 PAY0 PAY1 XOR`（6 字节，STM32 → Jetson）
  - 下行帧：`BB STATUS XOR`（3 字节，Jetson → STM32）
  - XOR 校验机制
  - 超时容错设计（30s 超时按正常放行）
- 4.5 PWM 电机控制与调速
  - TIM2 参数配置（Prescaler=71, Period=2999, 约 333Hz PWM）
  - TIM2 参数速查表

### 第七章 开发难点与解决方案

- 7.1 STM32F103C6 内存溢出问题
  - FreeRTOS heap 与链接脚本 RAM 区域的冲突分析
  - `._user_heap_stack` 溢出 104 字节的根因：`configTOTAL_HEAP_SIZE` 过大
  - 解决方案：精确计算最大可用 heap = RAM总量 − 其他.bss/.data − 链接器预留 = 5016B
  - 最终取值 4800B，保留 216B 余量
- 7.2 CubeMX 代码重新生成导致 PWM 参数丢失
  - Period 从 999→99、OCMode 从 PWM2→PWM1 的连锁影响
  - Compare=500 超出 Period=99 范围，输出恒为高电平
  - 解决方案：手动恢复 tim.c 原始参数，建立参数对照文档
- 7.3 `osKernelStart()` 后死代码问题
  - FreeRTOS 调度器启动后永不返回
  - PWM 启动代码误放在 `osKernelStart()` 之后导致电机不转
  - 解决方案：将初始化代码移至 `osKernelStart()` 之前的 `USER CODE BEGIN 2` 区域
- 7.4 步进电机高频失步
  - 上电直接输出高频 PWM 导致堵转的机理
  - 起步频率 ≤ 1kHz 的经验法则
  - 调速公式：速度 ∝ 频率 = 1MHz ÷ (Period + 1)
- 7.5 Jetson Nano 串口配置
  - nvgetty 占用 ttyTHS1 的解决方案
  - dialout 用户组权限配置
  - 回环测试验证方法
